# NuClaw 教程大纲 V2 - 完全渐进式设计

## 设计原则

1. **每一章解决前一章的问题** - 自然的演进动力
2. **每一章都有可运行的成果** - 学习有成就感
3. **知识点不跳跃** - 新概念都有前置铺垫
4. **代码量控制** - 单章不超过 600 行，模块化章节不超过 800 行

---

## 第一阶段：网络基础（3章）

### Step 0: 最简单的 Echo 服务器
**目标**：让程序能接收和响应网络请求
**代码量**：~60行
**解决的问题**：网络通信是什么？

**内容结构**：
1. **为什么学网络？** - 从本地程序到网络程序
2. **Socket 基础** - 操作系统提供的网络接口
3. **阻塞 I/O** - 最简单的网络通信模型
4. **实现 Echo 服务器** - 接收什么就返回什么
5. **测试运行** - 用 curl/浏览器访问

**本章成果**：
```
浏览器 → http://localhost:8080/hello
服务器 → 响应 "Echo: hello"
```

**为下一章铺垫**：
- 当前问题：一次只能处理一个连接
- 下一章解决：同时处理多个连接

---

### Step 1: 异步 I/O 与多连接
**目标**：让服务器能同时服务多个客户端
**代码量**：~120行
**解决的问题**：Step 0 的阻塞问题

**内容结构**：
1. **Step 0 的问题回顾** - 阻塞在单个连接上
2. **多线程方案** - 每个连接一个线程（问题：资源消耗大）
3. **异步 I/O 概念** - 非阻塞、事件驱动
4. **Boost.Asio 基础** - io_context、async_accept、async_read
5. **Session 模式** - 每个连接一个对象，但共享线程池
6. **CMake 构建** - 项目结构规范化

**本章成果**：
```
多个客户端同时连接，都能收到响应
```

**关键衔接**：
- 从"一个客户端"到"多个客户端"
- 引入异步概念（后续 Agent 的并发基础）

---

### Step 2: 应用层协议 - 从 TCP 到 HTTP
**目标**：理解协议分层，实现完整 HTTP 解析
**代码量**：~180行
**解决的问题**：Step 1 只有 TCP，没有应用层协议

**内容结构**：
1. **协议分层回顾** - TCP 是传输层，需要应用层协议
2. **HTTP 协议详解** - 请求行、头部、空行、Body
3. **请求解析器** - 解析 Method、Path、Headers
4. **响应构造器** - 构造状态行、Content-Length
5. **路由系统** - 根据 Path 分发到不同处理函数
6. **静态文件服务** - 返回 HTML/CSS/JS 文件

**本章成果**：
```
GET /index.html → 返回文件内容
GET /api/time   → 返回当前时间
GET /           → 返回 "Hello, World!"
```

**为下一章铺垫**：
- 当前问题：HTTP 是请求-响应模式，服务器不能主动推送
- 下一章解决：WebSocket 实现双向通信

---

## 第二阶段：实时通信（2章）

### Step 3: WebSocket 实时通信
**目标**：实现服务器主动推送，为 AI 对话铺垫
**代码量**：~250行
**解决的问题**：Step 2 HTTP 的单向限制

**内容结构**：
1. **HTTP 的局限性** - 为什么需要 WebSocket？
   - 对比：聊天软件用 HTTP 会是什么样？
2. **WebSocket 握手** - Upgrade 机制、101 Switching Protocols
3. **帧协议** - 文本帧、二进制帧、掩码
4. **实现 WebSocket 服务器** - 基于 Step 2 的 HTTP 服务器扩展
5. **双向通信测试** - 浏览器用 JavaScript 连接

**本章成果**：
```javascript
// 浏览器端
ws = new WebSocket("ws://localhost:8080");
ws.send("你好");  // 发送给服务器
ws.onmessage = (e) => console.log(e.data);  // 接收服务器推送
```

**关键衔接**：
- 现在有了"实时对话"的基础设施
- 但服务器回复是固定的（Echo）
- 下一章：让回复变得"智能"

---

### Step 4: 第一个"AI" - 从固定回复到规则回复
**目标**：引入"智能"概念，即使规则-based 也是进步
**代码量**：~320行
**解决的问题**：Step 3 只是 Echo，没有处理能力

**内容结构**：
1. **什么是智能？** - 从固定规则到理解意图
2. **简单的模式匹配** - 关键词 → 回复模板
   ```
   "你好" → "你好！很高兴见到你"
   "时间" → 返回当前时间
   "天气" → "请告诉我城市名"
   ```
3. **正则表达式匹配** - 更灵活的模式
4. **上下文记忆** - 记住用户刚才说了什么
   ```
   用户：北京天气如何？
   AI：北京今天晴天
   用户：那上海呢？（AI 知道是问天气）
   ```
5. **会话管理** - 每个用户有独立的对话上下文

**本章成果**：
```
用户：你好
AI：你好！有什么可以帮你的？

用户：现在几点
AI：当前时间是 14:30

用户：北京天气
AI：北京今天晴天，25°C

用户：那上海呢？
AI：上海今天多云，22°C（理解"那"指代天气）
```

**关键衔接**：
- 规则系统能处理固定场景
- 但规则太多会爆炸，且无法理解复杂语义
- 下一章：引入 LLM 解决理解问题

---

## 第三阶段：LLM 集成（2章）

### Step 5: 接入大语言模型（LLM）
**目标**：接入真实的 AI 能力
**代码量**：~400行
**解决的问题**：Step 4 的规则太死板

**内容结构**：
1. **为什么需要 LLM？** - 规则系统的局限
   - 无法处理没见过的问题
   - 维护成本高
2. **LLM 基础概念** - 什么是 GPT？Token 是什么？
3. **OpenAI API 详解** - HTTP POST、JSON 格式、API Key
4. **HTTP 客户端实现** - 用 Boost.Beast 发送 POST 请求
5. **Prompt 工程入门** - 系统提示、用户消息、温度参数
6. **流式响应处理** - 逐步显示（可选，或留到 Step 6）

**本章成果**：
```
用户：请用一首诗描述春天
AI：（调用 OpenAI API）
"春风拂面柳丝长，
桃花盛开满园香..."
```

**关键衔接**：
- 现在有了 LLM，能回答各种问题
- 但 LLM 不知道实时信息（天气、股价）
- 下一章：让 LLM 能使用工具获取实时数据

---

### Step 6: Function Calling - LLM 使用工具
**目标**：让 LLM 能调用外部工具
**代码量**：~500行
**解决的问题**：Step 5 LLM 知识是静态的

**内容结构**：
1. **LLM 的局限** - 知识截止、无法获取实时信息
2. **工具调用概念** - Function Calling 机制
   - LLM 不直接回答，而是决定调用什么工具
   ```json
   {
     "tool": "get_weather",
     "arguments": {"city": "北京"}
   }
   ```
3. **硬编码实现** - 第一个工具：天气查询（mock）
4. **工具执行流程** - 接收调用 → 执行工具 → 返回结果 → LLM 生成回复
5. **多轮工具调用** - 一个任务需要多个工具

**本章成果**：
```
用户：北京天气如何？
AI（内部）：调用 get_weather(city="北京")
工具返回：{"temp": 25, "weather": "晴天"}
AI：北京今天晴天，气温25°C

用户：那适合穿什么？
AI（内部）：已有天气信息，直接回答
AI：天气很好，建议穿短袖
```

**关键衔接**：
- 硬编码工具能工作，但难以扩展
- 每加一个工具都要改主代码
- 下一章：设计可扩展的工具系统

---

## 第四阶段：工具系统（3章）

### Step 7: Tool 系统设计
**目标**：从硬编码到可扩展的架构
**代码量**：~600行（模块化）
**解决的问题**：Step 6 硬编码不可维护

**内容结构**：
1. **Step 6 的问题回顾** - 硬编码的痛点
2. **设计目标** - 开闭原则：对扩展开放，对修改关闭
3. **Tool 基类设计** - 抽象接口
4. **Schema 定义** - 描述工具的参数和用途
5. **注册表模式** - 自动发现工具
6. **参数校验** - 类型检查、必填项检查

**新增工具示例**：
- 计算器（数学计算）
- 文件读取（带安全限制）
- 系统信息（时间、CPU）

**本章成果**：
```cpp
// 新增工具只需一个文件，无需改主代码
class CalculatorTool : public Tool {
public:
    ToolSchema get_schema() override { ... }
    json::value execute(const json::object& args) override { ... }
};

// main.cpp 中自动注册
registry.register_tool(std::make_shared<CalculatorTool>());
```

**关键衔接**：
- 工具系统能工作了
- 但同步执行会阻塞
- 下一章：异步执行优化

---

### Step 8: 异步工具执行
**目标**：非阻塞执行，提升并发能力
**代码量**：~700行（模块化）
**解决的问题**：Step 7 同步执行的性能问题

**内容结构**：
1. **同步的问题** - 调用慢工具时整个 Agent 卡住
2. **异步编程模型** - 回调、Future/Promise
3. **std::async 和 std::future** - C++ 异步基础
4. **工具超时控制** - 防止工具无限执行
5. **并发限制** -  semaphore 控制同时执行的工具数
6. **结果回调** - 工具完成后通知 Agent

**本章成果**：
```
用户：同时查询北京和上海天气
AI：并行调用两个天气查询
总时间 = max(北京查询时间, 上海查询时间)
而不是 北京时间 + 上海时间
```

**关键衔接**：
- 异步性能好了
- 但安全风险：工具可以执行任意代码
- 下一章：安全沙箱

---

### Step 9: 工具安全与沙箱
**目标**：安全地执行不受信任的代码
**代码量**：~800行（模块化）
**解决的问题**：Step 8 缺乏安全控制

**内容结构**：
1. **安全风险分析** - 代码注入、路径遍历、SSRF
2. **HTTP 工具安全** - URL 白名单、内网保护
3. **文件工具安全** - 路径规范化、沙箱目录
4. **代码执行安全** - 黑名单过滤、超时、资源限制
5. **沙箱机制** - 容器化概念（简介）

**新增工具**：
- HTTP 客户端（带 SSRF 保护）
- Python 代码执行（带黑名单）

**本章成果**：
```python
# 用户试图执行危险代码
import os; os.system("rm -rf /")
# → 被黑名单拦截："Forbidden: import os"

# 用户试图访问内网
http_get(url="http://localhost/admin")
# → 被拦截："URL not in whitelist"
```

## 总结

### 演进路线

```
网络基础 → 实时通信 → AI 能力 → 工具系统 → 企业级 → 生产部署
  (3章)     (2章)     (2章)     (3章)      (3章)     (2章)
```

### 每章衔接关系

| 章节 | 本章解决 | 为下章铺垫 |
|:---|:---|:---|
| Step 0 | 网络通信基础 | 但只能处理一个连接 |
| Step 1 | 多连接并发 | 但只有 TCP，没有协议 |
| Step 2 | HTTP 协议 | 但服务器不能主动推送 |
| Step 3 | WebSocket 双向通信 | 但回复是固定的 |
| Step 4 | 规则-based 智能 | 但规则维护困难 |
| Step 5 | LLM 智能 | 但 LLM 无实时数据 |
| Step 6 | LLM 使用工具 | 但硬编码不可扩展 |
| Step 7 | 可扩展工具系统 | 但同步阻塞 |
| Step 8 | 异步执行 | 但无安全控制 |
| Step 9 | 安全沙箱 | 但数据在内存 |
| Step 10 | 数据持久化 | 但无多租户 |
| Step 11 | 租户隔离 | 但只有 WebSocket |
| Step 12 | IM 集成 | 但无认证 |
| Step 13 | 安全认证 | 但无部署方案 |
| Step 14 | 容器化部署 | 完成！ |

### 与 V1 的主要区别

1. **LLM 提前**：Step 5 就接入 LLM，不是最后才接入
2. **工具自然出现**：Step 6 是 LLM 需要工具，不是突然讲工具
3. **WebSocket 提前**：Step 3 引入，为后续对话铺垫
4. **Step 4 新增**：规则-based AI 作为 LLM 的前置概念
5. **每章有明确问题驱动**：不是功能堆砌，而是解决问题
